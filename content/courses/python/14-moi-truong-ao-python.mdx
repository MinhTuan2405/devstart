## Môi trường ảo Python là gì?

Môi trường ảo Python là một không gian riêng cho từng project, nơi bạn cài thư viện mà không làm ảnh hưởng đến project khác trên cùng máy. Khi học `venv python`, bạn đang học cách tách môi trường chạy để mỗi dự án có bộ package và phiên bản của riêng nó.

Với người mới, chỉ cần nhớ một quy tắc rất quan trọng: một project nên có một môi trường ảo riêng. Quy tắc này nghe đơn giản nhưng giúp bạn tránh rất nhiều lỗi khó chịu về sau, nhất là khi bắt đầu cài thư viện ngoài như `requests`, `pandas` hoặc `flask`.

## Cách tạo venv trong Python

Python có sẵn công cụ `venv`.

```bash
python -m venv venv
```

Lệnh trên sẽ tạo thư mục `venv` chứa môi trường ảo.

Kích hoạt môi trường ảo:

```bash
# Windows
venv\Scripts\activate

# macOS / Linux
source venv/bin/activate
```

Sau khi kích hoạt, bạn thường sẽ thấy tên môi trường xuất hiện trước dòng lệnh, ví dụ `(venv)`. Điều đó báo rằng mọi lệnh `python` và `pip` tiếp theo đang chạy trong môi trường hiện tại.

Nếu máy bạn có nhiều phiên bản Python, đôi khi lệnh có thể là:

```bash
python3 -m venv venv
```

Tên thư mục môi trường không bắt buộc phải là `venv`, nhưng đây là tên rất phổ biến và dễ nhận ra.

## Cách cài thư viện trong môi trường ảo Python

Khi venv đã được kích hoạt, mọi lệnh `pip install` sẽ chỉ tác động đến project hiện tại.

```bash
pip install requests
pip list
```

Bạn cũng có thể lưu danh sách thư viện bằng:

```bash
pip freeze > requirements.txt
```

Đây là bước rất hay dùng khi chia sẻ project cho người khác hoặc khi chuyển project sang máy mới. Người khác chỉ cần cài lại đúng các thư viện đã dùng bằng cách đọc file `requirements.txt`.

Ví dụ quy trình thường gặp là:

- tạo môi trường ảo
- kích hoạt môi trường
- cài thư viện cần dùng
- lưu danh sách package vào `requirements.txt`

Khi đó, project của bạn sẽ rõ ràng và dễ tái tạo hơn nhiều.

## `pip` và `venv` liên quan với nhau thế nào?

Người mới thường nhầm hai công cụ này. Hãy tách chúng ra rất rõ:

- `venv` dùng để tạo môi trường riêng
- `pip` dùng để cài package vào môi trường đó

Nói ngắn gọn, `venv` tạo ra chiếc hộp riêng, còn `pip` bỏ thư viện vào chiếc hộp đó. Nếu bạn không kích hoạt đúng môi trường, `pip` có thể cài nhầm package vào nơi khác.

## Vì sao nên dùng venv cho từng project?

- Tránh xung đột phiên bản thư viện.
- Dễ tái tạo môi trường trên máy khác.
- Giữ project sạch và dễ quản lý hơn.
- Giúp bạn biết chính xác project này phụ thuộc vào package nào.

Ví dụ, project A cần `requests` phiên bản này, project B cần phiên bản khác. Nếu không có môi trường ảo, bạn sẽ gặp rắc rối rất nhanh.

Một ví dụ thực tế:

- project học web cần `flask==3.x`
- project cũ của bạn lại cần phiên bản khác

Nếu cả hai cùng dùng chung môi trường hệ thống, việc nâng cấp hoặc hạ phiên bản có thể làm một project đang chạy tốt bỗng dưng lỗi. Môi trường ảo giải quyết vấn đề đó bằng cách cô lập package theo từng dự án.

## Cách làm việc an toàn với requirements.txt

Sau khi cài thư viện bằng pip, bạn nên tạo file `requirements.txt`.

```bash
pip freeze > requirements.txt
```

Khi quay lại project sau một thời gian hoặc gửi cho người khác, file này đóng vai trò như danh sách nguyên liệu. Người dùng có thể cài lại bằng lệnh:

```bash
pip install -r requirements.txt
```

Đây là thói quen rất quan trọng nếu bạn muốn làm việc chuyên nghiệp hơn với Python, kể cả khi project còn nhỏ.

## Những lỗi thường gặp với môi trường ảo Python

- Cài thư viện trước khi kích hoạt venv.
- Quên tạo `requirements.txt`.
- Commit cả thư mục `venv` lên git.
- Tưởng rằng đã vào môi trường ảo nhưng thực ra terminal đang dùng Python hệ thống.

Thông thường, thư mục `venv` nên được bỏ vào `.gitignore`.

Ngoài ra, người mới còn hay gặp các tình huống sau:

- tạo venv thành công nhưng quên kích hoạt
- dùng `pip` của hệ thống thay vì `pip` trong venv
- xóa môi trường ảo rồi quên cài lại package

Nếu bạn nghi ngờ mình đang ở nhầm môi trường, hãy kiểm tra lại dấu nhắc terminal hoặc chạy `pip list` để xem package hiện tại.

## Quy trình chuẩn cho một project Python mới

Khi bắt đầu một project mới, bạn có thể làm theo thứ tự sau:

```bash
python -m venv venv
source venv/bin/activate
pip install requests
pip freeze > requirements.txt
```

Nếu dùng Windows, bạn thay lệnh kích hoạt cho phù hợp. Chỉ cần lặp lại quy trình này vài lần, việc quản lý package Python sẽ trở thành phản xạ tự nhiên.

## Bài tập thực hành

Tạo một thư mục project mới, sau đó:

- Tạo môi trường ảo
- Kích hoạt môi trường ảo
- Cài thử một thư viện bất kỳ bằng `pip`
- Lưu lại bằng `requirements.txt`

Gợi ý:

```bash
python -m venv venv
```

Sau khi làm xong, hãy tự trả lời 3 câu hỏi:

- nếu không kích hoạt venv mà cài package thì chuyện gì xảy ra
- `requirements.txt` có tác dụng gì
- vì sao không nên đưa thư mục `venv` lên git

Nếu trả lời được ba câu này, bạn đã hiểu phần cốt lõi của môi trường ảo Python.

## Câu hỏi thường gặp về môi trường ảo Python

### Có bắt buộc phải dùng môi trường ảo không?

Không bắt buộc tuyệt đối, nhưng rất nên dùng, đặc biệt khi project có cài thư viện ngoài. Đây là thói quen gần như mặc định trong hầu hết project Python thực tế.

### `venv` và `pip` khác nhau thế nào?

`venv` dùng để tạo môi trường riêng. `pip` dùng để cài thư viện vào môi trường đó. Hai công cụ này thường đi cùng nhau trong hầu hết project Python.

### Có nên đặt tên môi trường ảo là `venv` không?

Có thể. Đây là cách đặt tên phổ biến và dễ nhận ra nhất. Bạn vẫn có thể dùng tên khác, nhưng `venv` thường là lựa chọn an toàn cho người mới.

### Khi nào cần tạo lại môi trường ảo?

Bạn có thể tạo lại khi môi trường bị lỗi, khi chuyển sang máy khác hoặc khi muốn làm sạch toàn bộ package rồi cài lại từ `requirements.txt`. Vì môi trường ảo có thể tạo lại dễ dàng nên đừng quá sợ việc xóa và dựng lại.

## Tóm tắt

Bạn đã hiểu môi trường ảo Python là gì, cách tạo `venv`, kích hoạt, cài package bằng `pip` và lưu phụ thuộc vào `requirements.txt`. Đây là nền tảng bắt buộc khi bắt đầu làm project có thư viện ngoài. Ở bài tiếp theo, chúng ta sẽ học **đọc ghi file trong Python** để làm việc với dữ liệu thật thay vì chỉ nhập tay.
